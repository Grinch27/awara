# Awara 模块化与查询体系重构总方案（2026-05-12）

## 1. 文档目的

本文档用于统一三类读者的执行语义：

1. 工程负责人：快速判断当前阶段是否可上线、可回滚、可追责。
2. 开发者：按任务切片直接落代码，不再反复猜测边界。
3. GPT/Agent：拿到上下文后可连续执行“改代码 -> 提交 -> 远端构建 -> 修错”闭环。

本文档替代旧版“主页聚合入口优先”方案，主轴为：

1. 首页不再依赖 home 聚合页作为主入口。
2. 默认入口改为 video/image/forum（由设置页选择）。
3. Search 头部风格向 EhViewer 靠齐（三行结构）。

## 2. 当前现状（本次修复后）

### 2.1 已落地能力

1. `:build-logic`、`:core:model`、`:data` 已拆分并持续作为基础模块。
2. `:feature:player`、`:feature:search` 已参与主流程。
3. `FeedQuery` / `FeedScope` / `FeedFilter` 已在首页与搜索链路可用。
4. 首页默认入口偏好从“首页分区”语义收敛为“默认入口页面”语义。
5. 搜索页头部已改为三行：
   - 第一行：返回键 + 搜索框
   - 第二行：标签（Video / Image）
   - 第三行：日期下拉

### 2.2 已修复问题

1. 设置入口出现 `Still in developing` 的根因已修复：
   - 根因：旧的 `home` 聚合页在内部回调中把 `setting/history/download` 当页面内分区处理，落入 `TodoStatus`。
   - 修复：移除 `home` 导航项，工具入口继续直接路由跳转。
2. 首页默认入口从 `subscription` 调整为 `video`，并保留 `image/forum` 作为设置可选项。
3. 设置文案已同步更新：不再使用“首页默认分区”措辞，改为“默认入口页面”。

### 2.3 当前风险点

1. `forum` 入口仍为占位态，若用户将其设为默认入口，仍可能看到开发中提示。
2. 搜索页底部仍保留增强交互（快速搜索、摘要区），与 EhViewer 极简风格尚未完全等形。
3. `user` 搜索能力仍存在于 VM/数据层，但主入口不再展示切换标签，后续需明确产品策略。

## 3. 目标架构（最终形态）

### 3.1 模块分层

1. `:app`
职责：Application、导航装配、DI 装配、发布配置。

2. `:core:model`
职责：领域模型、查询模型、跨 feature 可复用 UI 状态模型。

3. `:core:network`（下一阶段新增）
职责：Iwara API、OkHttp/Retrofit、鉴权头、传输与重试策略。

4. `:core:database`（下一阶段新增）
职责：Room、DAO、本地缓存与清理策略。

5. `:data`
职责：仓储实现、远端/本地聚合、参数映射、错误归一化。

6. `:ui`（按需新增）
职责：主题、通用组件、分页壳、状态壳。

7. `:feature:*`
职责：页面编排、ViewModel、feature 级模型映射、交互状态。

### 3.2 查询模型统一边界

统一入口对象：

1. `FeedQuery`
2. `FeedScope`
3. `FeedFilter`
4. `FeedSort`（建议补齐，替代裸字符串排序）

统一约束：

1. UI 层不直接拼 `Map<String, String>`。
2. 参数转换集中在 data 层 mapper。
3. legacy `FilterValue` 仅作过渡输入，不作长期模型。

## 4. 与 EhViewer / FreshRSS 的对齐策略

### 4.1 对齐 EhViewer 的工程实践

1. 先稳定 build-logic，再扩 feature。
2. 先抽共享边界，再改页面表现。
3. 迁移按“可编译 + 可回滚 + 可观察”的最小步推进。

### 4.2 借鉴 FreshRSS 的产品思路

1. 查询条件是资产，不是临时 UI 状态。
2. 结构化查询优先于按钮堆叠。
3. 导入导出与恢复能力应服务排障与迁移，而非仅服务设置页。

## 5. 迁移路线图（可执行）

### 阶段 A：导航与入口语义收敛（进行中）

目标：消除首页路由歧义与工具入口占位态。

任务：

1. 移除 `home` 聚合导航项，避免工具入口被当作页面内分区。
2. 设置页默认入口限制为 `video/image/forum`。
3. 设置相关文案改为“默认入口页面”。

验收：

1. 从抽屉进入设置不再出现 `Still in developing`。
2. 手机/平板布局均可直接落到 `video/image/forum`。

### 阶段 B：查询模型收敛（进行中）

目标：Index/Search 共用一套查询拼装与分页协议。

任务：

1. 提供统一 `FeedQueryMapper`（query -> API params）。
2. 首页与搜索页 VM 移除自管字符串拼装。
3. 错误态统一输出 `UiState.Error(messageText=...)`。

验收：

1. Index/Search 同筛选条件下请求参数一致。
2. 分页入口支持幂等重试，不出现重复页。

### 阶段 C：网络与数据库边界拆出（待开始）

目标：`data` 成为唯一聚合层，feature 不直连底层实现。

任务：

1. 新建 `:core:network` 并迁出网络配置。
2. 新建 `:core:database` 并迁出数据库定义与 DAO。
3. data 层补齐 repo 测试桩，减少页面级回归。

验收：

1. app 模块不再持有网络与数据库实现细节。
2. feature 模块仅依赖 data 接口/模型。

## 6. CI 驱动开发规范（强约束）

1. 不在本地构建 APK。
2. 每次修复只提交最小改动。
3. 使用 `gh workflow run` 触发后，必须 `gh run watch --interval 10 --exit-status`。
4. `gh run view --log-failed` 无输出时，降级到 `gh run view --job <job_id> --log`。
5. 每轮只修首个阻塞错误，避免回归面扩大。

## 7. 本地接口扫测规范（隐私优先）

1. 凭据不得写入仓库文件、脚本明文或日志文件。
2. 本地 curl 输出必须脱敏后再落盘。
3. 接口扫测仅在本地终端执行，不上传 CI artifact。
4. 记录“现象+接口路径+状态码”，不记录 token/cookie 明文。
5. 播放排障仅记录链路阶段失败点，不记录完整私密请求头。

建议排障顺序：

1. 登录与 cookie 获取。
2. 视频详情接口返回媒体源。
3. 媒体源 URL 可持续读取。
4. 播放器数据源是否携带必要请求头。
5. 失败时写入匿名结构化日志。

## 8. 代码规范约束

1. Kotlin/Compose：遵循 Google Android 风格，优先状态提升与显式副作用。
2. Shell：遵循 Google Shell Style，默认 `set -euo pipefail`，避免明文凭据。
3. Python：兼容 3.8，复杂类型可用 `Any` 或省略变量类型声明。
4. 提交说明：一句话说明“修复点 + 触发场景 + 结果”。

## 9. Agent 执行模板（Awesome Prompt）

```text
你是仓库内执行型工程代理，请严格执行以下闭环：
1) 读取当前失败 run 的首个阻塞错误。
2) 仅修改与首个错误相关的最小代码。
3) 本地只做静态检查与错误面检查，不做本地构建。
4) 提交并 push 到当前工作分支。
5) 触发 build-apk.yml。
6) 每 10 秒 watch 一次，直到 run 结束。
7) 若失败：优先抓失败日志摘要（run 级失败为空则切 job 级日志），回到步骤 1。
8) 若成功：输出变更清单、影响面、残余风险与下一步建议。

输出要求：
- 先给结论，再列文件与原因。
- 禁止输出账号、cookie、token 等敏感信息。
- 引用日志仅保留错误摘要，不贴完整请求头。
```

## 10. 里程碑与验收清单

M1（已完成）：

1. `feature/search` 接入主导航。
2. 搜索核心 VM 迁移并可运行。
3. build-apk.yml 多轮修复后恢复绿色。

M2（进行中）：

1. 首页入口从 home 聚合页切换为直接主内容入口。
2. 设置页默认入口语义与选项收敛。
3. 搜索页头部三行布局落地。

M3（待开始）：

1. `:core:network` / `:core:database` 模块拆分。
2. Iwara 全接口本地扫测脚本模板化（脱敏、禁落盘凭据）。
3. 播放失败链路诊断事件标准化。

## 11. 回滚预案

1. 若新导航策略导致入口异常，优先回滚“导航接入层”，不回滚底层查询模型。
2. 若查询参数映射变更引发请求异常，保留 legacy mapper 作为临时兜底。
3. 若 CI 连续两轮失败且错误不收敛，冻结新增需求，仅做编译恢复。

## 12. 下一步建议（执行顺序）

1. 先补 forum 最小可用页，避免默认入口落入占位态。
2. 再推进 FeedSort 类型化与 mapper 收敛。
3. 最后启动 Iwara 接口本地扫测模板，专项定位“视频可见但不可播”的链路断点。
